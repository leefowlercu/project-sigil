# PRD-0220: Sigil-Web New Run Dialog Specification

## Status

Accepted

## Context

The root `sigil-web` workspace exposes a `New Run` affordance in the selected
agent context pane. Operators need a centered dialog that can author a run
configuration either through a guided form or through raw YAML before the
workspace starts the run through the selected live app-server session.

## Goals

- Define the root-workspace new-run dialog surface for the selected agent.
- Define how form authoring and YAML authoring interact within the same dialog.
- Define the live run-start outcome and inline invalid-config failure behavior.

## Non-Goals

- Defining standalone new-run routes outside the root workspace.
- Defining richer run-template management beyond the initial example-backed
  preset.
- Defining follow-on run-stop or run-edit behavior.

## Dialog Contract

- The selected-agent `New Run` affordance in the root workspace MUST open a
  centered modal dialog owned by the `/` route.
- The dialog MUST remain scoped to the currently selected agent.
- The dialog header MUST show only `New Run`; it MUST NOT render the selected
  agent description sentence beneath the title.
- The dialog MUST provide `Form` and `YAML` tabs for the same run draft.
- The dialog MUST render the `Form` and `YAML` tabs as a compact header mode
  switch instead of a full-width primary control block.
- The form-authored draft MUST use `prompt` and `context` keys instead of
  `prompt_template` and `context_template`.
- The initial draft MUST keep the example-backed OpenRouter, guardrail, and
  accounting settings in the generated YAML while allowing v1 form editing for
  run name, prompt, context, system prompt append, provider, model, reasoning,
  and RLM depth.
- The run-name controls MUST render as a compact inline row that defaults to a
  three-word hyphen-delimited autogenerated slug, with regenerate and edit icon
  actions.
- While autogenerated mode is active, the run name MUST render as a read-only
  value with explicit actions to regenerate it or switch into custom editing.
- Switching into custom editing MUST keep the current autogenerated slug as
  the editable starting value.
- Returning to autogenerated mode from custom editing MUST replace the current
  run name with a fresh three-word hyphen-delimited slug.
- The `Prompt` and `Context` fields MUST appear immediately after the run-name
  row and before model configuration controls.
- The `Provider` and `Model` controls MUST share one form row within a unified
  model configuration band.
- The reasoning controls and RLM controls MUST render as inline grouped
  controls within the model configuration band, with dependent fields dimmed
  when their toggle is off.
- The `Reasoning Effort` control MUST be disabled while `Enable Reasoning` is
  off and enabled when `Enable Reasoning` is on.
- The `RLM Depth` control MUST be disabled while `Enable RLM` is off and
  enabled when `Enable RLM` is on.
- The `Context` field MUST remain a first-tier visible form field.
- The dialog MUST expose a collapsible `Advanced Settings` section below the
  model configuration band.
- The `System Prompt Append` field MUST live inside `Advanced Settings`,
  default empty, and render the requested placeholder and supporting-label
  copy.

## Form and YAML Synchronization Contract

- While the dialog remains in generated mode, form edits MUST regenerate the
  YAML draft shown in the `YAML` tab.
- The first manual edit in the `YAML` tab MUST promote the draft into manual
  YAML mode.
- Once the draft is in manual YAML mode, later form edits MUST NOT overwrite
  the YAML textarea.
- The dialog MUST expose a reset affordance that restores generated YAML from
  the current form values after manual YAML edits.

## Live Submission Contract

- The dialog MUST disable run submission when the selected agent connection is
  not ready.
- On submit, the root workspace MUST send the current YAML textarea content as
  the `run/start` request payload for the selected agent session.
- When the live app-server accepts the run config, the root workspace MUST add
  the new run to the selected agent workspace and make it the active run.
- When the live app-server rejects invalid YAML, the dialog MUST remain open
  and MUST surface the error inline.

## Acceptance Scenarios

### Scenario SCN-0000: Opens the selected-agent new-run dialog as a centered modal from the root workspace

Given the root workspace shows a selected agent panel  
When a user opens the selected agent's `New Run` affordance  
Then the root workspace shows the new-run dialog for that selected agent.

### Scenario SCN-0001: Regenerates run YAML from form edits while the dialog remains in generated mode

Given the selected agent's new-run dialog is open in generated mode  
When a user edits form-backed run configuration values  
Then the dialog's YAML draft updates to match the current form values.

### Scenario SCN-0002: Preserves manual YAML edits without back-populating from later form changes

Given the selected agent's new-run dialog has entered manual YAML mode  
When a user later edits form-backed values  
Then the dialog preserves the manual YAML draft instead of overwriting it from
the form.

### Scenario SCN-0003: Restores generated YAML when the operator resets the manual YAML draft

Given the selected agent's new-run dialog is in manual YAML mode  
When a user resets the YAML draft from the form  
Then the dialog replaces the manual YAML draft with the YAML generated from the
current form values.

### Scenario SCN-0004: Starts a new run from the root workspace and selects it after the live app-server accepts the run config

Given the selected root workspace agent is connected through a live ready
app-server session  
When a user submits the selected agent's new-run dialog with a valid run
config  
Then the root workspace adds that new run to the selected agent workspace and
selects it as the active run.

### Scenario SCN-0005: Surfaces live app-server invalid-run-config errors inline without closing the dialog

Given the selected root workspace agent is connected through a live ready
app-server session  
When a user submits invalid YAML through the selected agent's new-run dialog  
Then the dialog remains open and shows the invalid-config error inline.

### Scenario SCN-0006: Defaults the run name to a read-only autogenerated slug and allows customizing or restoring it

Given the selected agent's new-run dialog is open in generated mode  
When a user switches between autogenerated and custom run-name actions  
Then the dialog preserves the current autogenerated slug when entering custom
editing, enables direct editing in custom mode, and generates a fresh slug
when restoring autogenerated mode.

### Scenario SCN-0007: Enables reasoning effort and RLM depth only when their parent toggles are enabled

Given the selected agent's new-run dialog is open in generated mode  
When a user toggles `Enable Reasoning` and `Enable RLM`  
Then the dialog enables or disables `Reasoning Effort` and `RLM Depth`
according to the current state of their respective toggles.
